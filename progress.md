## 2026-02-15 - Critical Crash Bug Fixed (Enter to Select Session)

Fixed sidebar_tui-3wl: The TUI was crashing with "Broken pipe (os error 32)" when pressing Enter in the sidebar to select a session. Root cause: daemon.rs line 1057 was breaking the client handler loop after receiving `ClientMessage::Detach`, closing the connection. But the client in main.rs was trying to reuse the same connection to send `ClientMessage::Attach` for session switching. Fix: Removed `DaemonResponse::Detached` from the break condition in handle_client() - now only `ShuttingDown` breaks the loop. This allows the client to detach from one session and attach to another on the same connection. Added E2E test `test_session_selection_no_crash` that creates two sessions and verifies switching between them doesn't crash. All 73 tests pass (54 lib/bin + 17 E2E + 2 scaffold). No clippy warnings. Binary reinstalled. Closed sidebar_tui-3wl. Next agent should pick up one of: sidebar_tui-qd0 (color change), sidebar_tui-7ex (sidebar padding), or another P2 issue.

## 2026-02-15 - Hybrid Message Handling Research Complete

Completed sidebar_tui-dhj research: Analyzed the hybrid synchronization approach for live preview. Root cause confirmed: MessageReader buffers data in main async loop, but send_daemon_message uses blocking decode_message with read_exact() that bypasses the buffer, causing message interleaving when Preview messages are sent async. Solution: "drain-before-sync" pattern - before any sync operation, process all buffered messages via try_parse_buffered(), read remaining socket data with short timeout, then clear() any partial data. Implementation effort is low (~30 lines + 5-6 call sites). Helper methods already exist (clear, has_buffered_data, try_parse_buffered). Risk is low since sync ops immediately restore terminal state anyway. Created detailed spike analysis at references/hybrid_sync_spike.md. Closed sidebar_tui-dhj. Next agent should pick up sidebar_tui-hp0 (separate sockets) or sidebar_tui-dcr (fully async) research to compare approaches before implementation planning.

## 2026-02-15 - Separate Socket Channels Research Complete

Completed sidebar_tui-hp0 research: Analyzed the separate socket channels approach for live preview. Examined two options: (A) True dual Unix sockets with handshake protocol and client ID tracking, (B) Single socket with internal mpsc channels for routing. Studied Zellij's patterns - they use crossbeam mpsc channels internally rather than multiple sockets; their web client uses dual WebSocket channels (control_channel_tx + terminal_channel_tx) but Unix IPC stays single-socket. Option A (dual sockets) has high complexity due to coordination overhead. Option B (internal channels) is medium complexity but cleaner than dual sockets. Compared to hybrid drain-before-sync (sidebar_tui-dhj), both socket approaches are more complex for marginal benefit. Created detailed spike at references/separate_sockets_spike.md. Closed sidebar_tui-hp0. Next agent should pick up sidebar_tui-dcr (fully async) research to complete the comparison.
