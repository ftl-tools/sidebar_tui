# Homebrew Tap and AUR Packaging References

Research for implementing Homebrew tap and AUR packaging for `sidebar-tui` (`sb` binary).

## Summary of Findings

### Homebrew Tap Strategy

For binary distribution (not source-build), you need a **private tap** (e.g., `username/homebrew-tap`
on GitHub) because `Homebrew/homebrew-core` does not accept pre-built binaries from third parties.

Two formula styles are available:

**Style 1: `if OS.mac?` + `if Hardware::CPU.arm?` blocks (cargo-dist style)**

This is the pattern used by cargo-dist and the axodotdev tap. The `if/if` nesting is cleaner:

```ruby
class SidebarTui < Formula
  if OS.mac?
    if Hardware::CPU.arm?
      url "https://github.com/..."
      sha256 "..."
    end
    if Hardware::CPU.intel?
      url "https://github.com/..."
      sha256 "..."
    end
  end
  if OS.linux?
    if Hardware::CPU.arm?
      url "https://github.com/..."
      sha256 "..."
    end
    if Hardware::CPU.intel?
      url "https://github.com/..."
      sha256 "..."
    end
  end
  def install
    bin.install "sb"
  end
end
```

**Style 2: `on_macos { on_arm { ... } }` DSL (flyctl/GoReleaser style)**

This is the modern DSL used by GoReleaser and superfly/flyctl. Slightly more verbose but the
standard output of GoReleaser's homebrew tap generator.

See: `flyctl-tap.rb` for a complete real-world example.

**Key limitation of `mislav/bump-homebrew-formula-action`:** It cannot update formulas that
use `if/else` conditions for different download URLs per arch (which is exactly what we need).
For a binary tap with per-arch downloads, the formula must be **fully regenerated** on each
release (not just bumped), or you need to use cargo-dist which manages the tap itself.

### Recommended Automation Approach

**Option A (Recommended): cargo-dist**

cargo-dist is the purpose-built tool for exactly this workflow. It:
1. Builds cross-platform binaries
2. Generates the tap formula with per-arch binary URLs
3. Pushes the updated formula to your `homebrew-tap` repo on release

Setup steps:
1. Create `username/homebrew-tap` GitHub repo
2. Create GitHub PAT with `repo` + `workflow` scopes
3. Add as `HOMEBREW_TAP_TOKEN` secret
4. Add to `Cargo.toml`: `[package.metadata.dist]` with `installers = ["homebrew"]` and `tap = "username/homebrew-tap"`

See: `cargo-dist-homebrew-docs.md` and `cargo-dist-tap-formula-example.rb` for the formula output format.

**Option B: Custom script in release.yml**

Write a Python/bash script that:
1. Downloads the `.sha256` files from the GitHub Release
2. Generates the formula `.rb` from a template with the new version + checksums
3. Commits the updated formula to the tap repo using `HOMEBREW_TAP_TOKEN`

**Option C: `mislav/bump-homebrew-formula-action` (NOT suitable for binary taps)**

This action only works for formulas with a single `url` + `sha256` pair (source tarballs).
It explicitly cannot handle formulas with `if/else` arch-specific URLs. Starship uses this
action to update `homebrew-core` (not their own tap), and their formula is a source build.

### AUR Strategy

For AUR, create a `sidebar-tui-bin` package (the `-bin` suffix is the convention for
binary AUR packages that don't build from source).

The key files are:
- `PKGBUILD` - the build script
- `.SRCINFO` - generated from PKGBUILD via `makepkg --printsrcinfo`

For multi-arch (x86_64 + aarch64) support, use `source_x86_64` and `source_aarch64` arrays
with corresponding `sha256sums_x86_64` and `sha256sums_aarch64`.

See: `aur-gitui-bin-PKGBUILD` for a clean multi-arch binary AUR package example.

For GitHub Actions automation of AUR publishing, use `KSXGitHub/github-actions-deploy-aur`.
The workflow needs:
1. An AUR account with SSH key
2. The private key stored as `AUR_SSH_PRIVATE_KEY` GitHub secret
3. `AUR_USERNAME` and `AUR_EMAIL` secrets
4. A script that generates the PKGBUILD with the new version + checksums

See: `aur-deploy-workflow-sane-fmt.yml` (lines 622-676) for the `publish_aur_package` job.
See: `aur-sane-fmt-generate-pkgbuild.py3` for how to generate PKGBUILD dynamically.

## Release Archive Naming (current sidebar_tui release.yml)

The existing release workflow produces these assets:
- `sb-{VERSION}-aarch64-apple-darwin.tar.gz` (macOS arm64)
- `sb-{VERSION}-x86_64-apple-darwin.tar.gz` (macOS x86_64)
- `sb-{VERSION}-x86_64-unknown-linux-musl.tar.gz` (Linux x86_64)
- `sb-{VERSION}-aarch64-unknown-linux-musl.tar.gz` (Linux arm64)
- `sb-{VERSION}-x86_64-pc-windows-msvc.zip` (Windows x86_64)

Each archive contains the `sb` (or `sb.exe`) binary plus README.md and LICENSE.

## Key Decision Points for sidebar-tui

1. **`cargo-dist` vs. custom script for Homebrew**: cargo-dist is the most polished solution but requires adopting its full release pipeline. A custom Python script to regenerate the formula on each release is simpler and compatible with the existing `release.yml`.

2. **Binary name is `sb`**: The AUR package would be `sidebar-tui-bin` with `provides=('sidebar-tui')` and the binary installs as `sb` (or possibly with a symlink/alias).

3. **One-time setup required**: AUR account with SSH key must be done manually. Homebrew tap GitHub repo must be created manually.

4. **The `mislav/bump-homebrew-formula-action` limitation**: This action is what starship uses, but starship submits PRs to `homebrew-core` with a source build formula. For a binary tap with per-arch downloads, it won't work.
